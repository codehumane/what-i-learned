
# 무엇이 1등 팀을 만드는가?

> Leading Effective Engineering Teams

## CH 01. 무엇이 효과적인 팀을 만드는가?

### 1.2 동기부여가 성과를 이끌어낸다

- 내적 동기부여 중요성 이야기
- 여기에는 자율성, 전문성, 목적의식 필요 (다니엘 H. 핑크, 드라이브 인용)
- 데이비드라는 엔지니어를, 수익 보다는 사회적 영향력에 목표를 둔 프로젝트로 이동시켜, 식었던 열정을 되살리기 (목적의식 사례)
- 팀원에게 개별 모듈 개발을 주도하게 해서 자율성과 주인의식 높이도록 (자율성 사례)
- 팀원에게 최신 도구를 접할 수 있게 해 업무에 필요한 기술을 능숙하게 하도록 (전문성 사례)
- 팀원의 기여가 조직 목표에 어떻게 직접적으로 영향을 미치는지 명확히 설명 (목적의식 사례)

### 1.3 효과적인 팀 만들기

#### 1.3.1 적합한 인재 모으기

**적절하게 우선순위를 정하고 독립적으로 업무를 수행**

- 조직과 프로젝트의 전략적 우선순위를 명확하게 공유
- 이를 통해 팀원들이 업무 우선순위를 설정하는 데 도움을 줄 수 있음
- 정기적 체크인 미팅과 목표 설정 회의를 통해 꾸준히 도움을 주고,
- 엔지니어들이 각자 맡은 업무를 효과적으로 처리할 수 있도록 필요한 자원과 지침을 언제나 활용할 수 있도록 준비

**다양성과 포용성**

중요하고 당연한 얘기들 계손 나오는 가운데, 아래 사례가 최근의 고민 중인 내용이라 기록.

> 하지만 초기 회의는 대화보다는 독백에 가까웠습니다. 똑똑하지만 경험이 부족한 신참들은 의견을 말하길 주저했고, 겨우 낸 의견도 경험 많은 이들의 목소리에 묻혀 머리기도 했습니다. 또한, 그들에게 "선배 엔지니어들이 말씀하시는 것들에 비하자면 제가 말하는 내용들은 중요하지 않은 것 같습니다." 같은 말을 주기적으로 듣기도 했습니다. 저는 이런 문제를 해결하기 위한 방안을 마련해야 했습니다.

이를 해결하기 위해 아래와 같은 방안을 만들어 진행했다고 함.

1. 라운드 로빈 회의 도입 (모두가 동등하게 자신의 생각과 의견 말하는 시간 할당)
2. 격주로 가상의 '문화 교류' 모임 진행 (각자의 문화에만 존재하는 독특한 것들을 공유하며 상호 존중의 바탕 마련)
3. 열린포럼을 열어 심리적 안전감 증진 (익명으로 생각과 걱정을 포스팅하고, 매주 팀 미팅에서 사소한 의견도 경청하고 고민함 -> 내성적인 사람들도 점점 아이디어를 용기있게 냄)
4. 신참 팀원과 베테랑 멘토 짝지어, 조직의 역학 관계 알려주고 업계 규범을 가르치고 경력 개발을 위한 소프트 스킬도 키움

#### 1.3.2 팀 정신 일깨우기

- 집단이 공동의 목표를 달성하기 위해 노력하고 서로 도울 때 팀 정신을 가졌다고 표현
- 이를 위해 3가지가 필요한데, 그 중 하나는 팀 내 역할과 책임을 명확히 하기
  - 사례로, 프로젝트에서 개발자들에게 문서와 단위 테스트를 직접 작성해야 함을 명시해 문제 개선
- 또 하나 필요한 건 공동의 목적
  - 목표는 팀이 달성해야 하는 것
  - 목적은 목표를 왜 달성해야 하는지에 대한 것
  - 아키텍트는 확장성과 성능, 개발자는 마감 기한과 코드 효율성, 디자이너는 미학과 사용자 경험에 대한 목표를 가짐
  - 만약 공동의 목적이 없으면 갈등, 사용성 문제, 은근히 늘어나는 업무 범위, 기술 부채 등을 마주하게 됨
  - 목적을 위해서는 프로젝트 목적, 대상 사용자, 기대하는 영향력을 명확하게 설명해야 함
  - 또한 주기적으로 각자의 전문성과 우려사항을 나누며 정기적인 체크인이나 워크샵 진행
- 다음으로 필요한 건 팀원 간의 신뢰 구축
  - 지식을 나누고 비난 없이 서로 도움을 요청해서, 문제가 있더라도 신속하게 해 나가고 사기가 높음
  - 또 다른 한 팀은 베테랑 개발자들로 구성되었지만 서로 신경 쓰지 않고 맡은 모듈에만 집중하여 협업 어려움
  - 신뢰를 위해 열린 소통과 피드백 주고 받기 장려, 서로 개인적으로 알아갈 기회 제공, 프로젝트 목표와 일정 투명하게 공유, 팀워크와 협업 장려

#### 1.3.3 효과적으로 리더십 발휘하기

리더 책임으로 아래 3가지 이야기.

1. 팀에 영감을 불어넣기
2. 영향력 행사하기
3. 공동의 목표를 향하도록 안내하기

그리고 팀의 성과와 그 성과가 비즈니스에 미치는 영향에 대해 조직의 내외부 이해관계자들과 공유하는 활동으로 전략적 가시성<sup>strategic visibility</sup> 소개.

1. 더 큰 목표에 일치시키기 - 팀의 업무를 조직의 주요 목표와 일치시키기 (조직의 프로젝트 중 팀의 역량을 잘 발휘할 수 있는 것에 집중한 사례 언급)
2. 팀의 고유한 강점을 이용 - 팀원의 고유한 기술과 지식을 파악 후 적극적 활용
3. 효과적으로 소통하기 - 정기적으로 진행 상황과 배운 것들을 공유 (투명성은 업무에서의 인정과 원활한 협력을 가져옴)

## CH 02. 효율성 vs 효과성 vs 생산성

### 2.1 효율성, 효과성, 생산성의 차이점

#### 2.1.1 목표

- 효율성은 낭비를 최소화하고 결과물을 극대화할 수 있도록 일을 올바르게 하는 것
- 단위 시간당 얼마나 많은 기능이 완전히 개발되고 테스트되었는가로 측정
- 효과성이란 올바른 일을 하고 적절한 성과를 내는 것을 의미
- 생산성은 효율성의 하위 개념으로 투입 대비 산출량을 측정
- 생산성은 결과물의 품질과 영향력을 고려하지는 않음
- 생산성은 종종 기계와 자본의 효율성을 측정하는 데 사용

| | 효율성 | 효과성 | 생산성 |
| -- | -- | -- | -- |
| 목표 | 일을 올바르게 수행하기 | 올바른 일을 하기 | 투입 대비 결과물 |
| 측정법 | 시간 | 고객 만족도 | 코드 줄 수 |
|      | 리소스 활용도 | 비즈니스 가치 | 기능 점수 부여 |
|      | 품질 | 사용자 채택률 | 스토리 포인트 부재 |

왜의 중요성에 관한 가상의 사례 이야기 소개.

- 회사의 주력 소프트웨어를 마이크로서비스와 쿠버네티스로 이전을 맡음
- 비즈니스에 중요한 작업을 하면서 새로운 기술을 배울 수 있는 기회로 설렘
- 분위기도 좋고 일정 문제 없이 스프린트 진행해 나감
- 하지만 사용자 피드백에서는 성능 개선도, 사용 편의성 향상도 없었음
- 데이터 기반의 리더는 지표 확인 후, 영향력 없는 기술의 왕국을 구축하고 있음을 알게 됨
- 더 빠르게 빌드하고 배포 수 있었지만, 사용자에게 도움 되는 기술적 고려사항은 활용하지 못함
- 효율성과 생산성에 대한 적절한 정의와 지표 개발에 집중해야 함

#### 2.1.2 측정 방법

생산성을 투자 대비 결과물을 측정

- 코드 줄 수
- 기능 점수 (소프트웨어의 복잡성과 제공하는 기능의 수)
- 스토리 포인트
- 데브옵스 지표 (리드 타임, 배포 빈도, 평균 복구 시간, 변경 실패율 등 개발 프로세스의 속도에 중점)

효율적인 소프트웨어 엔지니어링 방식은 시간이나 비용과 같은 자원의 낭비를 줄여줌. 효율적인 팀은 자원 사용을 최적화하여 생산성을 높이고 원하는 결과물을 만들어냄.

- 시간 (작업이 더 빨리 완료될수록 프로세스의 효율성이 높아짐)
- 리소스 활용 (시간, 비용, 인력 등의 자원을 더 잘 활용)
- 버그 수정률 (버그를 더 빨리 발견하고 수정)
- 결함 밀도 (코드 한 줄당 발견된 코드 결함의 수)
- 품질 (효율적일수록 고품질의 제품을 생산)

효과성은 개발 활동의 궁극적인 성과를 측정

- 고객 만족도
- 비즈니스 가치
- 사용자 채택률
- 투자자본수익률 (소프트웨어 개발에 소요된 총비용 대비 얻은 순이익을 기준으로 산정)
- 시장 출시 속도

#### 2.1.3 영향 요인

팀 규모

- 규모 큰 팀은 작업을 더 빨리 완료할 수 있지만, 의사소통·의견 조율·의사결정 위한 추가 노력 필요할 수도
- 규모 작은 팀은 작업 속도는 느릴 수 있지만 협업 원활하고 효율적일 수도
- 스크럼 등의 애자일 방법론은 팀의 업무량 최적 관리와 의사소통 병목 현상 해결 가능

다양성

- 다양한 기술은 팀의 효과성과 생산성 향상
- 그러나 지나친 다양성은 의사소통 어렵고 오해 가능
- 특히 서로 다른 가치로 논의할 때 더욱 문제가 될 수 있음
- 포용성 강화 필요

역할 명확성

- 모든 팀원은 팀 내에서 자신의 역할과 책임을 명확히 이해해야 함
- 자신의 업무가 무엇인지, 팀 내 역할이 무엇인지 이해하면, 불필요한 혼선과 중복 작업을 줄여 효율성과 생산성 향상 가능
- 업무 소유권이 모호하면 팀원 간의 갈등으로 이어져 시간과 노력이 낭비
- 명확성 확보는 팀원 간 집중력 향상

의사소통

- 원활한 의사소통 = 더 많이 질문하고 더 많이 논의
- 오인과 오해를 줄여 효율성을 높이고, 협업과 아이디어 공유를 촉진해 효과성 높임

기타

- 몇 가지 주제가 더 있었는데 특별한 내용 없어서 주제만 기록
- 업무 환경
- 도구 및 기술적 고려사항
- 코드 상태

### 2.2 성과 vs 결과물

- 효율성과 생산성은 **결과물**을 측정
- 효과성은 **성과**에 초점
- 결과물: 엔지니어링 작업으로 만들어진 산출물. 결과물은 어떤 작업이 진행되었거나 팀이 어떤 업무를 수행했음을 보여줌
- 성과: 수행한 업무의 실제 결과. 성과는 팀의 노력이 긍정적이고 가치 있는 변화를 가져왔다는 걸 보여줌

| 결과물 | 성과 |
| -- | -- |
| 새로운 애플리케이션 출시 | 더 많은 사용자를 확보하는 데 도움 |
| 코드 리팩터링 | 코드 성능 개선 |
| 새로운 기능 추가 | 사용자 경험 개선 |
| 디자인 문서 발행 | 개발 프로세스 간소화 |
| 새 API 배포 | 다른 기업과의 원활한 협업 진행 |
| 작업량 | 비즈니스 가치 |
| 속도 | 투자 비용 |
| 품질 | 사용자 채택률 |
| 정원 | |
| 코드 상태 | |

#### 2.2.1 결과물 및 성과 측정 방법

결과물은 수행한 활동에 따라 달라짐. 대개 수치화된 데이터로 측정 가능. 보고하고 검증하기 쉬움.

- 작업량: 생산된 제품의 수
- 속도: 제품 개발 과정의 속도
- 품질: 고객 기대 기준에 따라 식별한 결함 수
- 정원<sup>capacity</sup>: 프로젝트를 작업할 수 있는 엔지니어 수
- 코드 상태: 코드 리뷰 통해 측정한 기술 부채의 감소율

성과는 수행한 활동의 **영향력**에 따라 달라짐. 질적 측면과 양적 측면 모두 존재하므로 검증 더 어려움. 또한 서비스 이용자의 인식에 따라 크게 달라짐.

- 비즈니스 가치: 새로운 기능으로 인한 수익 또는 프로세스 개선으로 인한 비용 절감 등
- 투자 비용: 목표한 성과를 달성하기 위해 투입한 비용 (제품 개발을 위한 초기 투자 비용이 높으면 달성한 성과의 가치가 감소될 수 있음)
- 사용자 채택률: 수행한 작업 적용 후 전체 사용자 대비 신규 사용자의 비율

https://agilealliance.org/resources/experience-reports/actionable-metrics-siemens-health-services/

- 기존 Agile 지표인 스토리 포인트, velocity로는 실패
- 속도는 좋아보이지만 정작 일은 끝나지 않고, 일정 예측은 어려우며, 동시에 여러 작업 진행으로 병목 증가
- **많이 시작하지만, 완료되는 건 적음**
- 스크럼 중심에서 칸반 기반의 **흐름 중심** 사고로 전환
- 핵심 지표
  - **WIP**: 현재 진행 중인 작업 수
  - **Cycle Time**: 시작 - 완료 시간
  - **Throughput**: 일정 기간 동안 완료한 작업 수
- 특히 중요한 건 WIP 제한
  - Little's Law
  - WIP ↑ → Cycle Time ↓
  - WIP ↓ → Cycle Time ↑
  - **새 일을 시작하기 전에 기존 일을 끝내라**
- 이로 인해 Cycle Time 단축, 예측 가능성 향상, 품질 개선, 처리량 증가
  - velocity 대신 monte carlo 시뮬레이션으로 일정 예측 (85% 확률로 언제 끝날지 판단)
  - 또한 이해관계자에게 스토리 포인트 대신 시간 기반 지표인 Cycle Time으로 커뮤니케이션
- WIP 통해 데이터 기반 의사 결정 가능(병목이 어딘지 보이며, 개발자를 추가할지 테스트를 추가할지 등의 리소스 배분)

#### 2.2.2 결과물보다 성과에 집중하기

결과물을 간단하게 측정할 수 있다고 해서 결과물만 확인하는 **생산성 측정 함정**에는 아래 문제점이 있음.

- 정확한 측정 어려움: 리팩토링이나 버그 수정 등 다양한 작업이 있으며 어느 것이 특별히 중요하다고 판단하기 어려움
- 가치 없는 과잉 생산: 결과물이 좋은 성과로 이어지는지 여부와 관계 없이 결과물 찍어내는 데 집중되기도
- 합리적이지 않은 마감일: 비현실적 마감일은 완성도를 떨어뜨리고 오류를 늘리고 품질은 저하될 수 있음
- 번아웃: 결과물이 원하는 목표와 성과에 더 가까워지는 상황을 간과한채, 매일 압박과 경쟁속에 피로감 누적은 번아웃으로 이어질 수도

수박 효과도 언급.

> 결과물이 정해진 목표를 달성하여 표면적으로는 지표가 '초록색'이지만, 그 속을 ㄷ르여다보면 성과나 원하는 결과를 달성하지 못해 지표가 '빨간색'인 상황을 지칭

성과에 집중하기 위한 리더의 노력 필요.

- 비즈니스에서 목표하는 성과를 완전히 파악하고 명확하게 전달
- 이를 위해서는 비즈니스 인사이트와 사용자 조사 데이터가 필요
- 엔지니어링 팀을 이끄는 경우라면, **제품 팀이나 프로젝트에 관련된 비즈니스 이해관계자를 통해 지속적으로 조사 정보를 파악해야 함**
- 정보수집의 연결이 끊기게 되면 프로젝트 진행에 문제가 될 수 있음
- **성과에 집중해야, 팀이 목표를 향해 나아가고 있는지, 자원이 효과적으로 사용되고 있는지 파악할 수 있음**
- **또한, 팀의 체계나 프로세스에서 무엇이 효과가 있고 없는지 이해하고 이를 적절하게 조정하기 쉬워짐**

#### 2.3.2 절충안을 찾기 위한 노력

효율성이 효과성을 헤치지 않도록 다음 노력들 필요.

- 테스트 - 매우 효과적이지만 많은 시간과 자원이 드는 포괄적 테스트 -> 모든 테스트를 수행하진 않지만 빠르게 실행할 수 있는 더 효율적 테스트

- 또한 사용자 피드백 통해 사진이 있는 게스트 프로필, 목록, 예약 시스템에 집중
- 고급 검색 필터와 맞춤형 추천은 매력적 기능이었지만 당시 에어비엔비에겐 사치스러웠고 개발을 미룸

#### 2.3.4 효과성과 효율성 사이의 균형

- 결과물을 만들어내는 것과 성과에 우선순위를 두는 것 사이의 균형을 잡기 위한, 3가지 핵심 요인 정의
- 전략 — 올바른 것들을 파악하고, 그에 상응하는 목표를 도출하여, 목표 달성을 위한 전략 수립
- 지표 — 계획의 성공을 어떻게 측정할 것인가를 정하기 (이 과정에서 효율성이 효과성을 촉진함을 확인할 수 있음)
- 결단과 행동 — 계획을 끝까지 실행하고 완수하기

#### 2.3.5 일상에서 효과적으로 효율성을 달성하는 매일 매일의 팁

- 효과성에 우선순위 두기: 효율성과 생산성도 중요하지만, 언제나 올바른 일을 하는 것을 우선순위에 두는 습관
- 성과에 집중: 결과물이나 산출물에 근거한 생산성 측정 < 성과와 비즈니스 가치에 집중
  - **팀원 각자의 업무가 목표에 어떤 영향을 끼치는지 모든 팀원이 알고 있어야 함**
- 지표 활용하기: 올바른 것을 측정할 수 있다면, 효율성, 효과성, 생산성 사이의 균형 잡기에 도움
- 협업 장려하기: 협업은 지식, 기술, 아이디어를 나누고 효율성과 효과성 향상
- 팀원에게 권한 부여하기: 팀원이 의사결정을 내리고 자신의 업무에 주인의식 -> 효율성과 생산성 향상
- 끊임없는 발전

## CH 03. 효과적인 엔지니어링을 위한 3E 모델

엔지니어링 리더가 자신의 팀에 효과성을 철저하게 심어줄 수 있도록 도와줌.

1. 활성화(**E**nable) — 효과성 정의를 통해 효과성을 활성화 (정의 시, 효과성 측정 방법도 명시해야 함)
2. 권한과 자율성 부여(**E**mpower) — 효과성을 위한 전략을 잘 수행할 수 있도록 권한과 자율성 부여
3. 확장(**E**xpand) — 팀을 자립하게 만들어, 더 큰 팀이나 조직이 가진 문제 해결에 효과성 확장

### 3.1 활성화

- 성공을 위해서는 조직이 생각하는 성공이 무엇인지 파악해야 함
- 따라서 효과적인 엔지니어링을 위한 첫 단계는 비즈니스에서의 효과성이 의미하는 바를 정의해야 함
- 이를 효과성을 활성화한다고 표현

#### 3.1.1 비즈니스 유형과 팀의 크기에 따른 효과성 정의

조직이나 팀의 목표에 따라 효과성이 다름. 따라서 비즈니스 영역의 맥락을 고려해 효과성을 정의해야 함. 이를 위한 몇 가지 일반적 단계는 다음과 같음.

1. 팀의 목표와 과제 파악 (**매출 증가**, 고객 만족도 상승, 사회적 효과 극대화 등)
2. 성공을 측정하기 위한 관련 지표 결정 (매출 증가 → **수익 증가**, 신규 고객 유입률, 전환율 등)
3. 각 지표에 대한 목표 설정 (이 목표가 조직의 성과가 되며, 수익 증가 지표의 목표로는 예를 들면 **연간 수익 10% 증가**)
4. 성과 측면에서 효과성 정의 (설정한 목표를 달성한다면 효과적이라 할 수 있으며 목표를 달성하기 위한 효과성 설정)

다음은 효과성을 정의하는 데 도움이 될 팁.

1. 핵심 이해환계자를 정의 과정에 참여시키기 
2. 데이터와 분석 결과를 사용해서 정의를 뒷받침하기 (애플리케이션 성능 문제가 사용성에 영향을 준다면, **효과성의 정의에 성능 개선을 포함할 수** 있음)
3. 정의를 간결하고 명료하게 유지하기
4. 정의를 정기적으로 검토하고 필요에 따라 수정하기

아래는 몇 가지 예시들.

- 이윤 추구 기업의 효과성 — 수익성이나 투자자본수익률 극대화 관점의 정의
- 사회적 기업의 자금 모금이나 크라우드펀딩 플랫폼 엔지니어링 팀 — 도움을 받은 사람의 수, 조직 활동의 영향력, 대중 인식 개선, 참여 수준
- 같은 조직이더라도 팀 간 목표 충돌로 조율 필요할 수도 — 텔레그래프 광고 팀은 더 많은 광고 추가로 수익 vs. 핵심 사업과 엔지니어링 팀에게 광고는 성능과 사용자 경험에 악영향 가능

| 팀/조직 유형 | 효과성 지표 | 측정 지표 |
| -- | -- | -- |
| 스타트업 | MVP 출시 | MVP 출시되기까지의 시간 |
| | 비즈니스 주요 목표 달성을 위한 한정된 자원의 효율적 활용 | 경비 지출 속도와 자원 배분 효율성 |
| | 제품 잠재력과 시장 선도력에 기반한 자금 조달과 투자금 확보 | 조달된 자금과 투자 금액 |
| 중견 소프트웨어 개발 | 예산과 요구사항 만족하는 고품질 프로젝트의 시의적절한 제공 | 정시 출시 비율 |
| | 생산성과 프로젝트 성공률 극대화 위한 효율적 프로젝트 관리와 자원 배분 | 예산 준수, 프로젝트 수익성 |
| | 높은 고객 만족도, 장기적인 동반 관계 구축 | 고객 만족도 조사, 고객 유지율 |
| 오픈소스 커뮤니티 | 커뮤니티 회원들의 기여, 헌신, 피드백, 협력 | 활동하는 기여자 숫자, 기여도 개수 |
| | 목표 사용자 집단 내 소프트웨어의 광범위한 적용과 사용 | 사용자 채택, 다운로드 관련 통계 |
| | 커뮤니티 주도 피드백, 기여에 기반한 지속적 개발과 개선 | 릴리즈 주기, 커뮤니티가 제안한기능 요구사항 구현 건수 |

#### 3.1.2 효과성에 시동 걸기

효과성을 정의했으면, 팀에 명확하게 공유해야, 효과적으로 업무 수행하자고 할 때 함께 잘 이해할 수 있음. 유용한 방법으로 아래의 것들 소개.

- 의사소통: 팀 미팅이나 이메일 같은 다양한 방법으로 효과성의 정의를 전달
- 훈련과 교육: 효과성의 정의와 괄년된 훈련이나 교육 제공
- 측정: 효과성 정의에 기반해 팀원의 업무 진행 상황 점검
- 피드백: 마찬가지로, 효과성 정의 관점에서 팀원의 성과 피드백
- 보상과 안정

2020년에 가트너에서 조사한, 팀이 이해관계자에게 가치를 전달하고 그 가치에 걸맞는 목표 달성을 위한 3가지 핵심 요인 설명.

1. 팀이 표준을 정하도록 권한 부여: 비즈니스는 광범위하고 계속 변화. 리더는 팀에게 직접 표준을 만들도록 해야 함. 다만, 팀 효율성 위해 권고사항은 제공해주어야 함.
2. 핵심 기술 가다듬기: 목표 달성에 필요한 기술과 능력을 갖추도록 지원해야 함. 특히 다양성을 기르는 것을 강조.
3. 서번트 리더십 실천: 리더가 사전에 장애물을 파악하고 대처하는 게 효과적일 수도. 리더가 프로젝트 매니저, 거버넌스 파트너와 같은 이해관계자와 협력.

### 3.2 권한과 자율성 부여

- 효과성 정의를 통해 활성화 했다면 이제 권한과 자율 부여
- 필요한 지원, 자원, 자율성, 성장과 발전 기회 등을 제공
- 업무에 대한 소유권을 갖고 의사결정 내릴 수 있는 권한, 독립성, 신뢰 부여
- 자기 믿음을 키우고 성장을 촉진하며 책임감과 주인의식 고취

#### 3.2.1 기회는 살찌우고 문제는 굶기기

권한과 자율성 부여 이야기에서, "기회는 살찌우고 문제는 굶긴다?" 궁금. 일단 기회 살찌우기는 아래와 같음.

- 성공을 위해 필요한 자원, 지원, 기회를 제공
- ① 장점에 맞는 업무 분담
- ② 개인의 강점을 발굴/발전시킬 수 있도록 교육 환경 제공
- ③ 서로의 강점을 활용할 수 있는 팀 구성

문제 굶기기는 다음과 같음.

- 효과성을 저해하는 장애물, 문제, 비효율 영향력 최소화
- ① 감사, 피드백 통해 프로세스의 병목과 비효율 찾고 개선
- ② 팀 내 충돌, 문제를 즉각 처리
- ③ 업무의 효율적 수행을 위한 충분한 자원 확보해 일정 지연과 좌절감 방지

그 외에도 몇 가지 예시 제공.

- 지속적 배포와 피드백 순환고리: 더 자주 릴리즈함으로써 더 빠른 피드백 순환고리 구축
- 칸반 보드: '딱 필요한 시기에만'이라는 JIT, 수요가 있을 때만 작업한다는 pull 원칙에 기반. WIP 개수 한정 -> 과부하로 인한 문제나, 끝내지 못한 업무를 효과적으로 줄일 수 있음
- 위임, 권한과 자율성 부여: 권한과 자율성 -> 주인의식과 성장 추구, 동기 부여, 과도하게 세세한 관리 줄여 병목 줄임

#### 3.2.2 개인적 수준에서 효과성을 추구하기 위한 노력

저자의 개인적 문제 상황 이야기로 시작.

- 팀의 프로젝트와 인원이 크게 늘어남 -> 팀의 규모 > 저자가 팀의 모든 세부사항을 관리할 수 있는 수준
- 업무 협업 조율이 어렵고, 잦은 컨텍스트 스위칭이 발생하고, 업무 지식을 머릿속에 담아 두는 것도 어려움
- 이메일함에는 의사결정을 기다리는 끊임없는 요청들
- 쇄도하는 요청들은 지치게 하는 걸 넘어 업무를 점점 비생산적으로 만듦
- 내가 팀의 발전에 병목임을 깨달음

이 때 활용한 해결책들을 하나씩 소개.

① 회피수단이 아닌 도구로써 위임 활용
- 단순히 업무를 떠넘기는 것이 아니라 권한과 자율성을 부여하는 것
- 유일한 의사결정권자에서 멘토와 조언자로 역할을 전환
- 먼저 팀 리더들이 처리할 수 있는 일상적인 의사결정사항들 파악
- 각자의 새 역할에서 편안하고 효과적 업무 수행을 위해 정기적으로 점검과 조정
- RACI 차트(4장에서 소개)도 더 자주 활용

② 신뢰와 투명성에 기반한 문화 조성
- 위임은 백지에서 이뤄지지 않으며, 위임을 뒷받침할 환경이 필요
- 투명성 최우선 문화 조성 - 정기적 열린 포럼과 팀 회의를 통해 현재 진행 중인 프로젝트와 어려움을 인지
- 신뢰 - 팀원들의 결정을 신뢰하고, 팀원들은 실수를 저지르더라도 여전히 믿고 지지해 줄 것임이 바탕

③ 지휘통제를 능가하는 프로세스 최적화 활용
- 전통적 지휘통제 모델은 팀이 처한 역동적 환경에 적합치 않았음
- 유연성과 지속적 발전을 강조하는 애자일 방법론 도입
- 전세계에 흩어져 있는 팀 간의 협업을 위한 도구와 실천법도 도입

이러한 조치로 인한 변화.

- 팀원 리더십 능력 성장
- 팀원들의 자신감 향상
- 팀원들의 비즈니스 이해도 깊어짐
- 저자는 소방수 역할에서 벗어나 전략적 목표에 집중
- 전략: 새로운 기술 검토, 협력 관계 증진, 장기적인 제품 계획 수립 같은 것들
- 팀 업무를 꼼꼼하게 관리하는 것을 넘어 팀의 성공을 위한 방향을 정하는 것으로 변모
- 팀의 구조가 더욱 회복력 있고 동적(급격한 시장 변화에 대응하는 능력 향상)으로 변화

피터 드러커의 [5 Habits to Acquire to be Effective Executive](https://www.biggerplate.com/mindmaps/JjSWagZB/the-effective-executive-by-peter-drucker) 소개.

① know where **time** goes

- 이메일 확인이나 회의 참석이 필수긴 하나 너무 많은 시간 할애 X
- 핵심 역할에 제일 많이 기여하는 활동 파악, 영향 끼치고 싶은 영역 고민
- 팀과 조직에 제일 큰 가치를 가져오는 활동에 투자

② focus on what i can **contribute**

- 이건 책 내용이 다소 이상함
- 여기서 말하는 contribution은 결과에 관한 것
- **efforts to results > just busyness**
- 그냥 일하는 게 아니고 “나에게 어떤 성과가 기대되는가?”를 자문
- “내가 속한 조직의 성과와 결과에 중대한 영향을 미칠 수 있는 기여는 무엇인가?”
- output X outcome O

③ build on **strength**

- 자신의 강점, 동료의 강점, 팀의 강점에 집중하고 성과 내기
- 평균적인 만능형이나 약점 보완 < 강점 극대화

④ concentrate on the major area where **superior performance** will produce outstanding results

- 한 번에 한 가지 중요한 일에 몰두
- 전문성이 가장 큰 영향력을 발휘할 수 있는 핵심 분야 파악 필수
- 탁월한 성과 발휘 가능한 몇 가지 핵심 분야에 집중
- “지금 하는 일을 계속할 가치가 있는가?”를 계속 물어봐야 함
- "하지 않을 것" 정하기도 중요

⑤ make **effective decision**

- 의사결정, 절충안 마련은 엔지니어링과 리더십의 본질
- 관련 정보 수집하고, 선택지 비교 분석하고, 리스크 고려하고, 수집된 정보 기반한 결정을 내리기
- 팀원과 이해관계자들의 조언도 구하기
- 선택으로 인한 결과를 꾸준히 회고하고 교훈 바탕으로 의사결정 개선

#### 3.2.3 팀 효과성 모델 적용하기

- 팀 효과성 모델<sup>Team Effectiveness Model</sup>: 높은 성과를 거두는 팀을 만들기 위해 필요한 요인을 체계적으로 이해하고 평가하는 방법을 제시하는 프레임워크와 이론
- 렌시오니 모델과 터크만 모델을 소개
- 이 모델들의 흐름을 알면, 어떤 노력이 필요한지 파악하는 데 도움

① [렌시오니 모델](https://www.praxisframework.org/en/library/lencioni): 팀의 효과성을 해치는 5가지 함정.

1. 신뢰의 결핍
2. 충돌의 두려움: 신뢰가 없으면 의견 충돌이 두렵고 토론 대신 침묵 -> 나쁜 의사결정
3. 헌신의 결핍: 충돌의 두려움 -> 팀원 간 결속 약화 -> 의사결정 이행 사라지고 모호함 팽배
4. 책임의 회피: 헌신의 결핍 -> 책임감 붕괴 (결정에 동의하지 않은 팀원은 점점 더 행동에 책임감 느끼지 못함)
5. 결과에 대한 무관심: 책임 회피 -> 개인의 욕구를 집단보다 우선시 -> 팀 내 의사소통, 회사 전체의 성과 감소

② [터크만 모델](https://en.wikipedia.org/wiki/Tuckman's_stages_of_group_development): 일반적인 팀의 발전과 성숙 단계. 반드시 선형적인 건 아니고 여러 단계를 왕복하기도 하며, 새로운 팀원이 합류하면 격돈 단계로 후퇴하기도.

1. 형성 단계: 팀이 막 구성되고 어떻게 운영되는지 눈치 보는 단계
2. 격동 단계: 팀원들이 서로 알아가는 단계이나, 친숙해지면서 의견 다툼이나 인간관계 문제가 발생하기도
3. 규범 단계: 일정한 리듬을 찾아가며 안정적 운영 방식을 찾아감
4. 성과 단계: 서로 신뢰하고 존중
5. 해산 단계: 목표를 완수하고 해산, 서로의 성과를 돌아보고, 배운점을 공유하고, 성공을 축하, 인수인계

#### 3.2.4 효과성 극대화

- 적합한 도구와 프로세스 활용: 처리할 데이터의 양이 계속 늘어 dbt, 스노우플레이크 이전, 문서화 투자 등 진행
- 회의 축소 및 팀 보호: 가급적 불필요한 회의는 참석하지 않도록 보호
- 경영진의 지지와 자원 확보: 가치를 이해하고 지지하는 경영진을 확보하는 것이 중요
- 우수한 인력 채용과 인재 육성에 투자
- 팀의 성장과 효과적으로 규모 확장을 위한 계획 수립
- 기본적인 도구를 재발명 < 핵심 비즈니스 가치에 집중
- 자신의 진로와 우선순위에 대해 주의 깊게 고려

#### 3.2.5 레버리지가 높은 활동 파악하기

- 적은 노력, 시간, 돈으로 더 많은 성과 달성
- 개인적 레버리지: 도구, 기술, 지식
- 조직적 레버리지: 팀과 조직 구조

#### 3.2.6 구글에서 얻은 교훈

- 구글 엔지니어링 문화는 다양한 측면에서 구글러가 효과성을 달성할 수 있게 도움
- ① 표준화와 공유: 문제를 잘 해결하는 방법이 있으면 이를 내부적으로 광범위하게 활용해 효과 (공유 라이브러리, 도구, 프로토콜 버퍼, 맵리듀스, 빅데이블 등 도구와 추상화에 투자)
- ② 재사용: 재사용 가능한 교육 자료 개발에 엄청난 투자
- ③ 올바른 것을 자동화

### 3.3 확장

- 다양한 부서와 팀이 알아서 잘 운영되도록 자급자족하게 만드는 것
- 조직 전체의 효과성과 효율성을 향상시키기 위함
- 캐시는 여러 해 구글 크롬 부서의 다양한 팀을 잘 리딩
- 그러나 역할이 점점 커지면서 관리 효율성에 어려움
- 모든 부분을 여전히 세세히 챙기려다 병목이 됨
- 그래서 캐시가 자신의 팀을 더 신뢰하게 만드는 것부터 시작
- 어떤 업무들을 위임할 수 있을지 파악하고, 한 발 물러나 조언과 지원에 집중
- 물론 캐시의 전문성을 필요로 하는 의사결정은 수행
- 또한 수많은 보고서로 의사소통이 오히려 힘들었기에, 체계적 의사소통 채널을 만들고 정기적으로 간소화된 현황 회의 진행
- 팀원들이 자신의 업무 영역 내에서 스스로 결정 내리는 문화도 함께 조성 -> 캐시 부담도 줄고 팀의 자신감과 주인의식 향상
- 캐시의 업무 우선순위도 재조정 -> 세부적인 팀 운영보다는 전략 기획에 더 초점, 여러 프로젝트에 캐시가 어느 정도로 관여할지 범위 설정
- 팀의 리더들이 더 큰 역할을 맡을 수 있도록 멘토링도 진행 -> 든든한 2차 리더십 계층 육성
- 마지막으로, 관리 스타일을 스스로 돌아볼 수 있도록 정기적으로 팀으로부터 피드백 받기 시작

#### 3.3.1 리더십이 직면한 도전 과제

- 결국 사람이다: 인재 관리, 코칭, 올바른 사람이 올바른 역할을 맡도록 조정하는 업무에 더 많은 시간 할애
- 기술 전문성은 부차적인 이야기: 기술적인 역할을 해당 분야 전문가에게 맡기고, 조언하고, 방향을 제시하고, 다른 이들이 탁월함을 발휘할 환경 조성
- 영역이 확장됨에 따라 여러분의 자리는 사라진다: 영역은 점점 확대되고 복잡해지므로, 기존의 일들은 위임하고, 큰 틀에서 더 전략적이고 비전 있는 리더십 접근 필요
- 방해 요소와 복잡한 문제가 더 많아진다

#### 3.3.2 리더십의 영원한 세 가지 원칙

[SEAG의 6장](/book/seag/seag-culture.md#6-leading-at-scale) 이야기라고 함.

늘 결정하라

- ① **눈가리개 찾기**: 편견, 경험, 사용 가능한 정보의 한계 등의 정신적 장애물을 제거하고 문제의 해결책을 고루 고려하라는 이야기
- ② **핵심 절충안 파악**: 팀이 풍부한 정보에 근거해서 절충안을 의사결정하기
- ③ **결정하고 반복하기**: 결정을 내리고 실행했으면, 결과를 평가하고 다음을 위한 절충안을 조정해야 하며, 완벽한 해결책만을 기대하며 망설이지 말아야 함
- CRM 시스템을 사내에서 개발할지, 사용 솔루션을 도입해 커스텀할지 결정하는 사례도 소개하고 있음

## CH 04. 효과적인 관리 행동: 구글의 연구

- 옥시젠 프로젝트 소개와 적용 이야기
- 그 다음은 아리스토텔레스 프로젝트

### 4.1 옥시젠 프로젝트

- 훌륭한 매니저 자질 파악 목적
- 2008년 구글에서 수행

#### 4.1.1 짧은 역사 이야기

- 과거 구글은 중간 관리 계층 전부 없애기도 함
- 하지만 훌륭한 매니저가 팀 성과에 중요했음
- 훌륭한 매니저가 있는 팀이 행복도와 생산성 높았음 
- 매니저 중요성은 알았으니, 다음 질문이 나옴
- '무엇이 매니저를 훌륭하게 만드는가?'

#### 4.1.3 성과가 우수한 매니저의 행동

1. 좋은 코치로 행동하기
2. 과도하게 세세한 관리 대신 팀에 권한 부여하기
3. 포용적인 팀 환경을 조성하고 성공과 웰빙에 대한 관심 표현하기
4. 생산적이며 결과 지향적으로 행동하기
5. 경청과 정보 공유에 능숙한 의사소통 전문가 되기
6. 경력 개발 지원 및 성과 관리하기
7. 팀을 위한 명확한 비전과 전략 보유하기
8. 팀에 조언하기 위한 핵심 기술 능력 갖추기
9. 뛰어난 의사결정자
10. 회사 전반에 걸쳐 협업하기

##### 좋은 코치로 행동하기

- 건설적, 구체적, 시의적절, 심사숙고한 피드백
- 명확한, 이해하기 쉬운, 그대로 실행 가능한 업무 가이드와 기대사항 전달
- 정기적 1on1 통한 성과 조언과 경력 목표 상담
- 강점, 동기부여, 발전 필요한 영역에 대한 코칭
- 질문 통해 의견을 심사숙고할 기회를 제공
- 답변을 들으며 의견 근거에 대해 이해하기
- 안전하고 협조적 분위기 환경 조성
- 이를 통해 의견 표현, 문제 토론, 조언 구하도록
- 긍정 강화, 높은 기준 설정, 성공 축하, 성장 지향 사고방식 장려 등 동기 부여와 영감 불러오기
- 팀에 전파하고 싶은 특성과 행동을 몸소 보여주고 모범 보이기
- 진실성, 회복탄력성, 헌신, 성장 지향적 사고방식을 행동으로 보여주기

##### 과도하게 세세한 관리 대신 팀에 권한 부여하기

- **도전적인 과제로 업무 영역 넓히기**
- 신중하고 사려 깊게 개입하기
  - 자율성 부여와, 조언 등 의지가 되는 개입 사이의 균형
  - **예컨대 5일이 걸리는 작업을 맡겨도 매일 진행 상황 보고 X**
  - **대신, 업무 진행에 차질이 생긴다면 즉시 알려달라고 부탁**
- **자율성 장려하기**
  - 자주적으로 해결책 찾도록 허용
  - **꼭 필요한 경우에만 개입**
- 혁신과 신중한 위험 감수 장려하기
  - 창의적 생각을 하도록, 자신의 아이디어를 분석하거나 시험할 시간을 장려
  - 활동이 성공한다면 두둑히 보상해 신뢰 쌓기
- 팀의 치어리더 되기
  - 경영진, 다른 팀, 고객 등 외부 인사에 팀을 대변하고 옹호
  - 성과를 잘 전달하고 인정과 보상 받도록 하기
  - 공로에 대해 적절히 보상
- 건설적 피드백 제공
  - 성과에 대한 논의
  - 그리고 개선의 여지 있는 부분도 지원

##### 포용적인 팀 환경을 조성하고 성공과 웰빙에 대한 관심 표현하기

- **새 팀원이 환영 받는다는 느낌 갖게 만들기**
  - 짝이나 멘토를 배정하고
  - 인사를 나누는 자리 마련하고
  - 원활한 온보딩 위해 필요한 자원과 정보 제공
- 팀 내 유대감 키우기
  - 팀 빌딩 활동 장려
  - 서로 소통하고 친밀감 형성할 수 있는 기회 제공
- **팀을 지지하는 열정적 치어리더로 활약하기**
  - 팀의 성공을 축하하고
  - 개개인의 성취를 축하하고
  - 인정과 칭찬 등 긍정 피드백 제공
  - 긍정과 희망의 환경 조성 가능
- 예의범절을 몸소 보여주기
- 팀원들의 웰빙에 적극적으로 관심 가지고 이해하고 지원하기
  - 정기적 체크인 통해 업무 부담과 필요한 자원과 지원 제공
  - 적극적 경청 통해 공감대 나타내고 우려사항 해결
  - 충분히 지원받고 있다고 느끼는 환경 조성
- **좋은 시절과 나쁜 시절 가리지 않고 꾸준히 지원**
  - 업무 차질이나 실패의 힘든 시기에 팀 치어리딩
  - 비난보다 실수로 배우는 문화 조성
- 팀 내 심리적 안전감 조성
  - 열린 대화 장려
  - 다양한 관점 적극 수용
  - 문제나 갈등을 신속하고 공손하게 처리
- 간단한 것 하나<sup>One Simple Thing</sup>
  - 웰빙과 일과 생활의 균형 개선하는 간단한 1가지 정하기
  - 일주일에 세 번은 한 시간씩 운동하기 위해 휴식하기 등
  - 일주일 동안 모든 연락 끊고 휴가 다녀오기

##### 생산적이며 결과 지향적으로 행동하기

- 생산적이어도 결과는 얻을 수 있음
- 그러나 결과 지향적 태도가 필요
- **높은 기준을 세우고, 실제로 성과도 내야 함**
- **성공이 무엇인지 팀과 눈높이 맞추기**
- **어떤 방식으로 도달할지도 이야기**
- **이를 위해, 명확한 목표를 세우고, 정기 회의로 진행 상황 추적하고, 필요하면 업무 조율도 진행**
- **그리고, 팀의 결과에 대해 책임지기**
- 방법들로는 아래의 것들 이야기
- ① 다양성을 갖춘 팀 만들기
- ② **비전과 전략을 측정 가능한 목표로 변환하기**
  - 회사 전반의 비전과 전략을 명확하고 측정 가능한 팀 목표로 변환
  - 거대한 목표를 작고 달성 가능한 마일스톤으로 작게 나누어 목표를 이해하고 향해 나아갈 수 있게 하기
- ③ **팀을 체계화하고 목표 달성을 위해 자원 분배하기**
  - **팀의 체계와 구조는 효율성과 생산성 극대화에 중요**
  - 팀원의 기술과 전문성을 평가하고 그에 따라 업무와 역할 할당
  - 예산, 기술, 교육과 같은 필요한 자원도 확보
- ④ 누가 무엇을 담당하는지 명확하게 정리하기
  - **팀 내 업무와 책임을 깔끔하게 설정**
  - **혼란과 업무 중복을 방지**
  - **자신의 헌신이 팀 전체 성공에 이바지함을 알게 하기**
- ⑤ 팀 장애물 제거하기
  - 장애물이나 어려움을 적극 파악하고 제거
  - 프로세스로 인한 문제, 한정된 자원, 외부 요인으로 인한 문제들 사전 대비하고 해결
  - 문제 해결 지원, 조언 제공, 적절한 영향력 행사
- ⑥ 선견지명을 발휘하여 잠재적 위험에 대비하기
  - 조직의 여러 도전을 대비하기 위해 업계 동향, 시장 변동, 조직 내부 변화 지속 점검
  - 변화로 인한 잠재적 혼란에 대비하는 계획 미리 세우기
- ⑦ 경청과 정보 공유에 능숙한 의사소통 전문가 되기
  - 개방적 토론 장려
  - 언제나 관심, 호응
  - 리더에게 얻은 정보 공유하고 배경과 맥락도 설명

##### 경력 개발 지원 및 성과 관리하기

- 팀원에게 기대하는 바를 정직하고 투명하게 전달
- 그들의 발전을 위한 폭넓은 피드백 제공
- 건설적인 방식의 비판 제공
- 성과 평가 시기 외에도 일년 내내 주기적으로 피드백 제공
- 성장 기회 제공과 새로운 기술 학습과 성공적 경력 쌓기 지원
- ① **성과 기대치를 명확하게 소통**
- ② 직원의 성과를 공정하고 공평하게 평가
- ③ 성과와 보상의 연관성과 보상 체계 설명
- ④ 경력에서 승진 이외의 다양한 가능성에 대해 조언
  - 사다리 타고 올라가는 것만 있는 게 아님
  - 조직 내 수평 이동, 특별 프로젝트, 부서간 협력 기회 가능성
- ⑤ 팀원이 회사 내에서 성장하고 변화할 수 있는 방안 지원
- 구글에서는 **경력 개발에 GROW 활용**
- **G**oal: 무엇을 원하나, 개인의 경력 목표, 동기 부여, 가치관 확인
- **R**elaity: 현재 어떤 상황인가, 자신의 역할에 대한 어려움이나 좌절감 등의 감정 확인
- **O**ption: 무엇을 할 수 있을까, 목표에 더 가깝게 갈 수 있는 선택지 논의
- **W**ill: 이제 무엇을 할까, 최선의 선택지와 첫 발 내딛는 데 도움 되는 절차 파악

##### 팀을 위한 명확한 비전과 전략 보유하기

- ① 팀원에게 영감을 주는 비전과 전략 마련하기
- ② 팀과 회사의 비전과 전략을 일치시키기
- ③ 비전을 만드는 과정 중심에 팀원 두기
  - 팀원들의 의견, 아이디어, 그들의 관점에 대해 적극 질문하고 경청
  - 주인의식과 자율성 고취
  - 의사결정 과정에 팀을 참여시키기
  - 협업을 장려하며, 팀의 집단 지성이 반영된 공동의 비전 만들기
- ④ 비전을 명확하게 전달하여 팀의 이해를 돕고 질문을 장려하기
  - 전문용어 사용 피하고, 명확하고 간명한 언어 사용
  - 모두가 비전을 쉽게 이해할 수 있게 하기
  - 비전이 만들어진 배경과 맥락에 대해 설명
  - 명확히 이해할 수 있게 질문을 통해 의문점을 해소하는 기회도 제공
- ⑤ **팀의 전반적인 전략이 어떻게 팀원의 업무와 맞물려 돌아가는지 설명하기**

구글에서는 팀의 가치를 규정하고 이를 단기 목표와 연결하기 위해 아래 단계를 사용.

```
핵심 가치 ←→ 목적 (왜)
    ↘     ↙
      사명
       ↓
      전략
       ↓
      목표
```

- 핵심 가치: 팀이 믿는 신념들
- 목적: 팀이 존재하는 근본적 이유
- 사명: 팀이 달성하고자 하는 것
- 전략: 팀의 사명을 실현하기 위한 계획
- 목표: 전략 실행을 위한 단기적으로 달성 가능한 목표

##### 팀에 조언하기 위한 핵심 기술 능력 갖추기

- ① 팀이 기술적 복잡성을 헤쳐 나가도록 도와주기
  - 팀이 직면한 기술 복잡성을 이해
  - 팀원들의 어려움과 장애물을 극복할 수 있도록 조언
  - 명확성과 방향성 제시, 충분한 지원 제공, 실용적 해결책 찾도록 도움
- ② 업무에서 마주치는 도전 과제 이해하기
- ③ 기술적인 능력을 활용하여 문제 해결하기
- ④ 비즈니스 요구사항에 맞춰 새로운 기술 습득하기
- ⑤ **기술/비기술직군 이해관계자 사이의 간극 메우기**

##### 뛰어난 의사결정

- 가장 중요한 특성
- 기술 팀 매니저에게 특히 더 중요
- 팀이나 회사의 향후 방향성과 방침에 대한 결정을 내려야 하기 때문
- 여러 선택지 비교해 기술적 지식, 비즈니스 요구사항, 목표에 근거해 최선의 선택
- 다양한 각도에서 보고, 확보된 정보에 근거해 의사결정
- 올바른 의사결정을 위한 전략은 다음과 같음
- ① 비즈니스에서 최선의 이익을 염두에 두고 효율적으로 의사결정
  - 긴박감 있는 일에는 불필요한 지연 없도록 신속히 의사결정
  - 편견이나 주관적 감정 아닌 사실, 데이터, 논리적 추론에 기반
- ② 주어진 상황을 고려하여 의사결정 내리기
  - 각 상황만의 고유한 특성과 맥락 고려
  - 자원, 리스크, 기대하는 성과 포함해 구체적으로
  - 상황의 복잡성, 시급성, 잠재적 영향력에 근거
- ③ 의사결정의 근거를 명확히 알려주기

##### 회사 전반에 걸쳐 협업하기

- 단순히 자신의 부서나 팀에만 초점 X
- 회사 전체를 보는 더 큰 관점 O
- 다른 집단과의 협업이 성공에 필수적임을 잘 알고 있기
- 원활한 부서 간 협업 위해 꾸준히 아래들 진행
- ① 부서 간 협업 필요할 때, 비즈니스 전반의 목표와 일치하는 공동의 목표와 성과에 높은 우선순위 매기기
- ② 전사적으로 최선의 결과를 가져다줄 수 있는 다른 팀과의 동반자 관계를 구축할 기회 찾기
  - 요청이나 지시 기다리지 않고, 팀이 적극적으로 전사 목표와 전략에 따라 업무를 진행할 기회 찾기
- ③ 다양한 팀과 부서 간의 협업 롤 모델 보여주기
- ④ 팀이 회사의 관행과 정책을 준수하도록 책임감 불어넣기
- ⑤ 회사의 문화와 커뮤니티에 적극적으로 참여하기

#### 4.1.4 성과

- 옥시젠은 매니저 중요성 입증
- 매니저가 가져야 할 본질적 특성을 정량화하고 제도적으로 확립
- 여기서 확인된 행동은 직원들의 성과, 만족도, 이직률에 긍정 영향
- 구글 매니저 수상 선발 기준도 옥시젠 프로젝트의 핵심 행동을 반영

#### 4.1.5 옥시젠 프로젝트의 연구 결과 활용하기

- 우선, 자신의 배경과 현재 보유한 기술 파악에 활용
- 연구 결과를 체크리스트 삼아, 부족한 영역과 수준 확인
- 교육과 발전 기회 탐색 및 보완
- 나머지 내용도 특별한 게 없어 이 부분 기록은 여기서 마무리

### 4.2 아리스토텔레스 프로젝트

- 팀의 효과성에 기여하는 주요 요인 파악
- 성공하는 팀을 만들기 위한 다섯 가지 주요 원동력 밝힘
- 이제부터는 매니저가 어떻게 이 원동력을 팀에 자리잡게 만들지에 대한 이야기

| 요소       | 설명                              |
| --------- | -------------------------------- |
| 영향력      | 자신의 일이 가치 있고 변화를 만든다고 느낌 |
| 의미       | 개인적으로 중요한 일이라고 느낌          |
| 체계·명확성  | 역할, 계획, 목표가 명확함              |
| 신뢰성      | 약속을 지키고 높은 기준을 충족           |
| 심리적 안전감 | 자유롭게 의견을 내고 실수를 인정하며 서로 도움 |

#### 4.2.1 심리적 안전감

- 대인 관계에서 안전하다는 믿음
- 아리스토텔레스 프로젝트에서 가장 중요한 요인
- 집단 응집력(구성원들이 잘 지내는 것) != 심리적 안전감(자유로운 발언도 안전)
- 성과 기준 완화가 심리적 안전감을 높이는 건 아님
- 마찬가지로, 부적절한 행동을 용인해서도 안 됨
- 심리적 안전감을 높이는 방법은 다음과 같음
- ① 경쟁자가 아닌 협력자로 갈등에 접근하기
  - 갈등을 성장과 학습의 기회로 삼기
  - 편을 들거나 비난 대신,
  - 존중 기반의 열린 토론 마련,
  - 각자의 관점을 자유룝게 표현하도록 장려
- ② 사람 대 사람으로 대화하기
  - 사람 대 사람의 대화로 관계와 신뢰 형성
- ③ 반응을 예상하고 대책 세우기
  - 민감하거나 까다로운 주제 논의에 앞서 잠재적 반응이나 저항 예상
  - 미리 준비해서 더 나은 대답, 문제나 두려움을 지원과 이해심으로 완화
  - 이런 준비는 세심함과 공감 능력을 드러내며 긍정 기여
- ④ 비난을 호기심으로 전환하기
  - 실수나 문제 발생 시, 비난 대신 호기심
  - 문제의 근본 원인이나 내막을 깊게 파악, 교훈 찾기
  - 성장과 학습의 기회
- ⑤ 소통 방식에 대한 피드백 요청하기
- ⑥ 심리적 안전감 측정하기
  - 익명 설문조사, 포커스 그룹, 면담 등 활용
  - 정기적 심리적 안전감 평가와 측정은 필수

#### 4.2.2 신뢰성

- 개인이 아니라 팀으로써 성취하려면 서로 신뢰하고 협력
- 팀원뿐만 아니라 리더도 올바른 일을 할 것이라는 믿음 있어야 함
- 회의 시간을 지키는 단순한 일도 신뢰가 보임
- 신뢰할 수 있는 팀원과 리더는 아래 특성을 보임
- ① 진정성 가진 의도
  - 신뢰가 가는 사람은 견고한 윤리적 기반을 가짐
  - 이들의 의도는 진정성을 가짐
  - 개인적 야망보다 팀의 공동 목표 우선
  - 팀과 팀원의 이익이 극대화되는 행동 통해 헌신을 보임
  - 진실, 정직, 공정 등의 가치 기반 행동으로 신뢰를 받음
- ② 책임감
  - 신뢰의 기본 특성
  - 업무에 대한 주인의식 갖고, 행동과 성과에 책임
  - 마감 기한 준수, 약속 이행, 고품질 결과물 제공
  - 솔선수범, 문제에 주도적 대처, 해결책 찾아 팀 성공에 기여
- ③ 합리적 사고방식
  - 비판적이고 분석적 추론
  - 논리적이고 사려 깊은 사고
  - 연관 정보 따지고, 다양한 시각 고려
- ④ 꾸준한 기여
  - 팀 업무에 꾸준히 기여
  - 토론, 회의, 프로젝트 적극 참여
  - 아이디어, 통찰력 제공
  - 자신만의 높은 기준 설정
- ⑤ 본보기로서의 리더십
- ⑥ 협업과 상호 의존성 강화
- ⑦ 역할과 기대치를 명확히 정의
  - 개인의 역할, 책임, 기대 성과에 대해 투명하게 소통
  - 자신의 업무와 기대치를 명확하게 파악할 수 있다면 책임감 높아지고 효과적 수행
- ⑧ 열린 소통 장려
- ⑨ 도움이 되는 피드백 제공

#### 4.2.3 체계와 명확성

> 중요한 결론 중 하나는 '구글에서는 누가 팀원인지보다는 팀원들이 어떻게 상호작용하고 업무의 체계를 잡고 자신의 기여를 어떻게 보느냐가 훨씬 더 중요하다'입니다.

- 체계화: 팀원이 자신에게 기대하는 바를 파악하게 함
- 목표/계획 명확성: 올바른 방향으로의 업무 진행에 도움
- 팀원은 팀의 체계와 프로세스 안에서 위치선정할 수 있어야 함
- 각자가 맡은 영역과 상호 기대사항이 분명해야 함
- 그래야 반복적인 방향성 논의 없이도 자율적으로 올바른 방향으로 일할 수 있음

##### 목표 및 핵심 결과(OKR)

- 목표에 대한 명확성 확보와 방향 설정에 OKR
- 무엇을 달성할지, 성공은 어떻게 측정할지 정의
- 팀의 OKR은 회사의 전반적 목표와 일반적으로 일치
- 분기별 OKR은 조직 OKR 달성에 영향 큰 것들을 선정

```
목표: 제품 수익 증대 촉진
핵심 결과:
  - 모든 사용자를 대상으로 X 기능 출시
  - 사용자당 수익률 Y퍼센트 증가를 위해 X 프로젝트 시행
  - 수익 성장을 견인하는 요인이 무엇인지 파악하기 위해 세 가지 수익 관련 실험 시작하기
  - 1분기에 Y기능을 구현하기 위한 기술 지원 확보

목표: 제품의 평판 개선
핵심 결과:
  - 세 개의 업계 행사에서 발표함으로써 제품의 입지 재확립하기
  - 상위 X명의 사용자를 파악하여 직접 연락하기
  - 사용자가 제보한 오류에 대한 응답 시간을 XX퍼센트 줄이기
```

##### RACI 차트

- 책임 할당 매트릭스
- 체계와 명확성을 가져오기 위함
- 간단한 격자로 이뤄진 표
- 사람들의 책임을 명확히 밝히고 팀에서 챙겨야 하는 모든 일들이 제대로 관리되는지 확인

#### 4.2.4 의미

- 여기서의 의미는 목적의식, 성취감, 발전을 경험하는 것
- 사람마다 다른 의미를 가짐 - 단순 코딩, 문제 해결 경험, 재정적 안정, 사회 변화, 가족 부양, ...
- 자신의 일에 의미와 목적 느낄 때 최선
- 그리고 자신의 기여가 중요함을 깨달았을 때 의미 찾기 쉬움
- 긍정적 피드백과 인정은 보통 도움이 됨
- 강력한 지원 역시 가치 인정과 존중의 또 다른 표현

#### 4.2.5 영향력

- 업무 중요성, 영향력 이해가 업무 성과에 도움
- 사소한 업무 하나도 조직 목표에 한 걸음 다가가는 것
- 다음은 팀원에게 왜 프로젝트가 중요하며 자신이 중요한 사람인지 전달하는 데 도움되는 방법
- ① 조직 목표와의 연결고리 만들기
  - 조직 목표와 각 팀원 업무 사이의 명확한 연결고리 제공
  - 목적의식과 동기부여에 도움
- ② 팀 비전을 향해 노력하기
  - 팀 비전을 함께 만들어가기
  - 주인의식과 일체감, 팀워크에 도움
- ③ 고객과 사용자에게 미치는 영향 이해
- ④ 실적과 성과 연결하기

## CH 05. 일반적인 효율성 안티패턴

- 엔지니어링 팀에서 발생하는 효율성 안티 패턴
- 개인적, 관행적, 구조적, 리더십 관련 안티 패턴 다룰 예정
- 가볍게 기록

### 5.1 안티패턴 분류

개인적

- 특정 팀원들이 보이는 특정 성향이나 행동에서 비롯
- 예컨대 다른 사람을 지나치게 돕는 팀원은 팀에 의존성 문제.
- 안티패턴에 책임 있는 사람이 바로잡아야 함

관행적

- 프로세스 내에서 발생하는 문제 관행
- 작업 흐름, 방법론, 의사소통 과정 등의 결함
- 팀 리더가 수정하고 결정하고 소통해야 함

구조적

- 팀 구성, 역할 할당, 협업 방식 등의 구조 문제
- 최적화되지 않은 팀 구조에 의한 지식 사일로, 의사소통 단절, 불균형 기술 분배 등이 여기에 해당
- 여기서는 대응 방안이 명시적으로 나오진 않음

리더십

- 리더의 행동, 결정, 조치에서 비롯
- 너무 세세하게 관리하려는 리더 등을 가리킴
- 리더 혹은 고참이나 상급자 개입 필요

### 5.2 개인적 안티패턴

-  개인의 특정 접근 방식이 처음엔 효율적 & 결과지향적
-  그러나 결과적으로 팀 역학에 부정 영향 줄 수도

#### 5.2.1 스페셜리스트

- 여러 스프린트 동안 같은 모듈에 배치
- 문제를 점점 빠르게 해결하고 생산성 높아짐
- 하지만 개인이 해당 영역의 전문가가 되는 수준이 지나치게 됨
- 모든 전문가에게는 전문가의 60-80% 수준의 지식 백업 인력 있어야 함
- 전문성 과잉 위험을 완화하고, 장기적으로 팀의 전문성이 지속 가능해야 함
- 숙련가를 키우면서도 서로 효과적으로 협업하고 지원할 수 있는 팀이어야 함

#### 5.2.2 제너럴리스트

- 엔지니어가 자신의 기술을 다각화하고 다양한 곳에 기여하려는 욕구에서 발생
- 과도하게 넓고 얇게 일을 벌임 → 깊이와 전문성 ↓
- 심층적 이해 부재와 피상적 역량으로 이어짐
- 팀 효율성 하락, 복잡하거나 중요 작업 처리에 문제
- 책에서는 T자형을 이상적이라고 소개
- 팀은 융통성과 다양한 목적을 위해,
- 일반적 지식을 풍부히 갖추면서도,
- 특정 영역의 깊이 있는 전문적 능력을 갖춘 사람들로 구성

#### 5.2.3 오래 붙들고 있다가 마지막에 공유

- 중간 공유가 잘 안 됨
- 팀에는 거대한 PR을 마지막에 공유
- 피드백 주기가 느려져 문제 조기 발견과 수정이 어려움
- 코드 리뷰는 병목이 되거나 제대로 리뷰 되지 못함
- 다른 팀원들은 작업에 대한 정보를 알지 못함
- 따라서 도움 될 통찰력 제공이나 효과적 협업 기회와 시기 놓침
- 팀의 기민함과 응집력 저하
- 투명성과 지속적 통합 지향 문화 필요
- 대응: 작은 PR 자주 제출, 스탠드업 미팅, 초기 설계 공유, draft PR, 짧은 중간 데모, WIP 공유도 안전하다는 분위기 조성

#### 5.2.4 끈질긴 조언자

- 처음엔 좋은 의도로 다른 사람들 지원하고 코드 품질 개선에 도움
- 그러나 팀이 여기에 지나치게 의존하고 스스로 결정 내리지 못하면 문제
- 해결책으로 아래 이야기들 나옴
- 도움 요청 전에 독립적으로 문제 해결 장려
- 서로 다른 수준의 경험을 가진 사람들 짝지어 학습 독려
- 학습 장려 위해 정기적 지식 공유 세션 마련
- 조언자에게는 자신의 업무 집중 시간 할애 권유
- 조언자에게 도전적 업무 맡기기 (익숙하지 않고 시간 많이 필요한 업무)

#### 5.2.4 사소한 개선자

- 작고 의미 없는 변경을 지속적으로 수행
- 리팩토링이나 재구성이라는 구실로 작고 중요치 않은 수준의 코드 변경 시도
- 비효율적이고, 개인의 성장과 성취 기회 놓침
- 상대적으로 의미 약한 곳에 집중력 소모
- 사소한 개선자에게는 도전적 작업 할당해, 기존 코드 수정 대신 뭔가 만들어 내게 하기
- 리팩터링 전에 변경의 잠재적 영향을 신중하게 검토하도록 장려
- 피상적 수정 X 실질적으로 도움 되고 의미 있는 개선 중시 환경 조성 O
- 프로젝트 목표에 부합하도록 코드 변경에 대한 명확한 기준 설정

### 5.3 관행에서 비롯된 안티패턴

- 개인에게 책임을 물을 수 없는 절차가 가지는 함정의 문제
- 리더가 팀 전체와 함께 표준 마련과 절차 준수 필요

#### 5.3.1 최후의 영웅담<sup>Last-minute heroics</sup>

출시 직전에 어려움 겪고 있을 때 영웅적 방식으로 해결. 조언자들이 나서고 팀원들은 뼈를 갈아 겨우 대응. 회자되고 칭송 받지만 매 번 상황 반복. 이는 여러 문제를 가져옴.

1. 시간 부족 → 확인/피드백 부족 → 문제를 놓침
2. 임시 방편들 → 기술 부채 쌓임
3. 결국 품질 악화 → 고객 만족/신뢰 저하
4. 영웅 의존 → 조직 스케일링에 불리, 병목

잘 정의된 프로세스는 아래와 같음.

1. 개발 주기 전반에 걸친 꾸준하고 반복적인 계획, 검토, 테스트
2. 열린 소통 장려 통해 장애물 조기 파악
3. 중요/핵심 문제 잘 다룰 수 있게 백로그에 우선순위 관리
4. 결승성만 바라보는 경주마 대신 지속 가능한 페이스 유지

#### 5.3.2 종잡을 수 없는 PR 프로세스

PR 리뷰 과정에서 직면하는 다음의 문제들을 가리킴.

1. 충분한 검토 없이 빠르기만 한 승인 → 위험성 증가, 코드 품질 아쉬움
2. 셀프 머지 → 이해 상충 위험, 다양한 관점 고려 부족 → 오류 수정이나 개선 여지 줄어듦
3. 장수 PR → 의사소통과 결정에 비효율성 → 개발 주기 느려짐
4. 마감 직전의 PR → 일정 초반의 충분한 PR 리뷰 대비 위험

아래 고려가 필요함.

1. 철저한 코드 리뷰 진행과 책임 발휘 환경 마련
2. 다른 사람의 머지를 받도록 규칙 시스템화
3. PR 방치 예방 절차 도입 (막연한 합의로 충분할지 고민됨)
4. 진행 상황 점검과 병목 파악 위한 체크포인트 설정
   - 일일 스탠드업 미팅, 일정 지연 원인이나 요구사항 변경 논의하는 고정 주기 회의

##### 구글에서의 코드 리뷰

- 코드 리뷰에서 기대하는 바: **교육, 규칙 지키기, 게이트키핑, 사고 예방**
- 보통 개발자는 일주일에 3개 정도의 변경 작성
- 작은 변경에 대한 첫 피드백은 보통 **1시간** 이내로 전달
- 큰 변경에 대한 첫 피드백 도착은 대략 **5시간** 이내
- 변경 사항 1개당 검토자 수의 중간값은 **1명**이고
- 1명 이상이 필요한 크고 복잡한 변경은 전체의 **25%** 미만
- 구글에 새로 합류한 사람에게 코멘트가 더 많음
- 오래된 직원은 100줄 당 보통 4개의 코멘트로 안정화
- 프로세스 붕괴는 리뷰 중 상호작용에서 발생
  - 변경 필요 이유에 대한 오해 등

#### 5.3.3 장기간에 걸친 리팩터링

- 1명 이상 엔지니어가 투입되는 리팩터링이 오랫동안 지속
- 예상보다 더 오래 걸리고, 완벽주의 혹은 부족한 지식으로 초기 의도와 달리 길게 늘어짐
- 리팩토링의 의도치 않은 범위 확장, 일정 지연, 투입 인원 추가, 집중력 분산 등은 문제
- 만약, 리팩터링이 장기화되면 원인을 파악해야 함
- 완벽주의 문제라면 확실한 경계와 영역 설정
- 지식 부족에 의한 경우라면 정보를 공유하거나 관련 경험 있는 엔지니어 연결
- 현실적인 시간 제한도 도움
- 정기적 동료 검토, 마무리 시점 정하기
- 열린 소통으로 프로젝트가 목표를 향해 갈 수 있도록 돕기

#### 5.3.4 적당히 넘어가는 회고

회고는 끊임없는 발전의 밑바탕이고, 팀의 원동력이며, 혁신을 이끔. 그러나 아래의 경우 효과 적음.

- 충분한 시간 투자 없으면 심도 있는 문제 분석 어려움
- 바빠서 회고를 건너뛰면 기회 자체를 상실
- 너무 늦으면 세부사항 기억 어려움
- 구조와 체계가 없다면 난잡해짐
- 소수만 얘기 혹은 반대 의견 제시 피하기는 위험
- 후속 조치가 없다면 책임감 떨어지고 회고는 형식적
- 근본 원인 탐구 없이 겉핥기 증상 처리는 회고 목적 달성 X

아래 노력 필요.

- 프로젝트의 반복 주기로, 혹은 주요 단계마다 회고 시간 배정
- 충분한 시간 투자로 심도 있는 탐구와 논의
- 아래 프레임워크 등을 활용한 체계적 접근
  - [시작/중지/계속 프레임워크](https://www.betterup.com/blog/start-stop-continue)
  - [화남/슬픔/기쁨 프레임워크](https://miro.com/templates/mad-sad-glad-retrospective/)
- 심리적으로 편안한 환경 조성으로, 팀원의 고른 참여와 다양한 의견 이끌기
- 회고 후, 조치사항 문서화하고, 책임자 선정하고, 다음 회고에서 진행 상황 점검
- 4 whys나 피쉬본 분석으로 근본 원인 파악
