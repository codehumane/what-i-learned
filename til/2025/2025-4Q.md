# 10.01

## 소프트웨어 엔지니어 가이드북

### 유용한 엔지니어링 프로세스

- 완료의 정의(DoD<sup>Ddefinition of done</sup>) 합의와 명시 필요.
- 린팅 규칙 따르지 않는 경우 PR 병합 차단(ADR 등의 정리에서 나아가 시스템화가 중요).
- 코드 리뷰 얘기도 나오는데, 이 역시 DoD가 필요하고, 관례와 규칙이 필요.
- 생산성과의 타협 위해 커밋 후 코드리뷰 방식을 취할 수도.
  - 리팩토링 커밋과 PR은 리뷰하지 않는다던 Tidy First의 책 내용도 생각남.
- 새로운 서비스 구축 생산성을 위한 스캐폴딩 시스템(여기에 언급될 정도인 게 인상 깊었음).
- `실험 후 소독`이란 이름을 사용하는데, 비활성화 된 피처 토글 자동으로 캡처하는 자동화 툴 구축 이야기.
  - 이를 제거하는 PR을 자동으로 만들어 엔지니어에게 제안할 수도.

# 10.30

## 적정 소프트웨어 아키텍처

적절한 제약 조건을 사용해 시스템이 의도하는 바를 계속 유지하는 방법에 대해 소개.

1. 판단을 구체화: 설계 제약 사항으로 지식을 직접 전달하지 않고도 수용 가능한 해결책을 다른 엔지니어가 만들 수 있게 함.
2. 개념적 무결성<sup>conceptual integrity</sup> 장려: 일관되게 적용한 좋은 단일 아이디어가 시스템 전체에 흩어진 여러 훌륭한 아이디어 보다 나음. 아키텍처 제약은 불필요한 창의성 소모를 감소시켜 필요한 곳에 창의성 사용 장려.
3. 복잡성 최소화: 예측 가능한 setter 동작, 데이터는 데이터베이스에만 저장 등과 같은 시스템을 이해하기 쉽게 만듦.
4. 런타임 동작 이해: 제약을 통해 런타임 동작 소스 코드의 예측 가능성을 높인다고 하는데 잘 와닿지는 않음.

# 11.09

## 리더의 생각

### 여러분은 주니어와 시니어를 어떻게 구분하나요

- 개인 성장 단계에 맞는 조직의 도움을 얘기하고 있음.
- 처음 경력 시작
  - 동기부여 매우 높고 실무 능력은 부족
  - 따라서, 구체적인 업무 지시, 과정 검토/피드백 필요
- 1-3년 경력 쌓은 후
  - 1차 침체기라고 함
  - 실무 역량은 올랐지만 동기부여는 낮아짐
  - 이직을 선택하기도 하지만 이 효과는 잠시 뿐이고 다시 침체
  - 문제 해결 위임으로 도전과 성취 제공
  - 단순 지시형이 아니라 적절한 업무 분배와 기술 의사결정 참여 기회 제공
- 1차 침체기 후
  - 역시나 권한 위임과 필요한 지원 제공 필요
  - 열린 질문으로 스스로 해결책을 도출할 수 있게 해야 함
  - 이 과정으로 스스로 의사결정할 수 있는 역량을 쌓고 동기 부여
- 마지막
  - 완전한 권한 위임이 가능한 단계
  - 전문 지식과 경험이 있으며, 문제를 정의하고 결과를 도출하며, 높은 동기 부여 상태 유지
  - 업무 영향력 확대를 위해 내/외부 지원을 얻을 수 있어야 하고, 동료 및 다른 팀과의 효과적인 협업 환경을 갖출 수 있어야 함
  - 기존에 경험치 못한 갈등 상황에 노출되기도
  - 이런 갈등을 해결하고 리더십 역량을 갖춰 나갈 수 있도록 지원
