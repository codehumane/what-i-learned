Siemens Health Services가 어떤 지표를 써야 실제로 개선 되는지를 실험한 사례.

https://agilealliance.org/resources/experience-reports/actionable-metrics-siemens-health-services/

- 기존 Agile 지표인 스토리 포인트, velocity로는 실패
- 속도는 좋아보이지만 정작 일은 끝나지 않고, 일정 예측은 어려우며, 동시에 여러 작업 진행으로 병목 증가
- **많이 시작하지만, 완료되는 건 적음**
- 스크럼 중심에서 칸반 기반의 **흐름 중심** 사고로 전환
- 핵심 지표
  - **WIP**: 현재 진행 중인 작업 수
  - **Cycle Time**: 시작 - 완료 시간
  - **Throughput**: 일정 기간 동안 완료한 작업 수
- 특히 중요한 건 WIP 제한
  - Little's Law
  - WIP ↑ → Cycle Time ↓
  - WIP ↓ → Cycle Time ↑
  - **새 일을 시작하기 전에 기존 일을 끝내라**
- 이로 인해 Cycle Time 단축, 예측 가능성 향상, 품질 개선, 처리량 증가
  - velocity 대신 monte carlo 시뮬레이션으로 일정 예측 (85% 확률로 언제 끝날지 판단)
  - 또한 이해관계자에게 스토리 포인트 대신 시간 기반 지표인 Cycle Time으로 커뮤니케이션
- WIP 통해 데이터 기반 의사 결정 가능(병목이 어딘지 보이며, 개발자를 추가할지 테스트를 추가할지 등의 리소스 배분)

## SEAG, Code Review

오랜만에 SEAG 책의 [코드리뷰](/book/seag/seag-processes.md#9-code-review) 내용 다시 살펴 봄.

## 코드 리뷰의 최종 목적

- 다른 엔지니어에게 변경에 대한 동의를 얻는 것
- LGTM = 내 이름 걸고 이 변경에 동의

## 리뷰의 3가지 검사

- 정확성, 표현력 검사: 저자가 주장한 동작을 하는지, 다른 엔지니어가 이해 가능한지 확인
- 코드 소유자의 승인: 구글 모노레포 특성에 기인한, 코드의 정확성/표현력을 넘어, 조직/아키텍처 관점 승인
- 가독성: 스타일 가이드, 베스트 프랙틱스 준수 확인

## 코드 리뷰 이점 6가지

- checks code correctness
- ensures the code change is comprehensible to other engineers
- enforces consistency across the codebase
- psychologically promotes team ownership
- enables knowledge sharing
- provides a historical record of the code review itself

### code correctness

- 주관성 방지를 위해 저자 접근 방식 존중
- 개인적 선호의 대안 X 표현력(복잡도 등)이나 기능성(효율 등)을 향상시키는 대안 O
- 완벽한 해결책 동의 X 코드베이스가 향상된다면 승인 O -> 리뷰 속도, 스케일 ⬆

### psychologically promotes team ownership

- 코드가 자신의 것이 아님을 인지하게 도움
- 리뷰 절차가 중립적이고 정제된 비판을 주고 받도록 돕기에, 비판의 감정적 완충 작용도 함
- 팀의 validation -> 안정감
- 리뷰 올리기 전 한 번 더 검토하게 됨

## Code Review Best Practices

- 코드리뷰가 마찰과 지연 가져올 수도 있으나
- 이는 코드리뷰 그 자체의 문제 X 어떤 코드리뷰 구현을 골랐느냐의 문제 O
- Be Polite and Professional
  - 리뷰어는 저자의 접근 방식을 존중, 결점이 있을 때만 지적
  - 다른 대안이 동일하게 유효하다면 저자의 선호 수용
  - 코드리뷰는 24시간 내에
  - 리뷰어는 나눠서 하지 말고 한 번에 (저자의 고통 줄이기)
  - PTAL
- Write Small Changes
  - 변경의 35%가 한 파일에 대한 것
  - 규모 큰 변경은 정당히 거절 가능
  - 다만 새로운 메이저 개발 등 때로는 작은 변경 강제가 오버헤드가 되기도
- Keep Reviewers to a Minimum
  - 보통 3가지 승인(구글엔 승인 유형이 3가지)을 1-2명이 담당
  - 첫 번째 LGTM 이후의 것들은 효용이 그렇게 커지지 않음
  - 리뷰어가 여럿일 경우가 있긴 하며, 이 때는 서로 다른 관점을 명확히 갖고 진행

## Types of Code Reviews

- 유형에 맞는 리뷰가 존재
- Greenfield Code Reviews
- Behavioral Changes, Improvements, and Optimizations
- Bug Fixes and Rollbacks
- Refactorings and Large-Scale Changes

### Moving from and Individual Contributor Role to a Leadership Role

- 선장이 없는 배는 그냥 물 위를 떠다니는 대기실
- **누군가 방향을 잡지 않으면, 의미 없이 시간을 보내거나, 심지어 필요 없는 코드를 만들게 됨**
- 코드 작성은 사과를 따는 일
- 관리는 바나나를 키우는 일
- 몇 개의 기능을 만들었느냐 대신,
- 팀이 행복감을 느끼고 생산성 있게 만들었는가?
- 그래서 하루 지나도 눈에 드러나는 결과물이 없어 보임

#### Servant Leadership

- 관리자가 되면 갑자기 '관리'병
- 마이크로매니징, 저성과자 외면, 말만 잘 듣는 사람 채용 등
- **"Above all, resist the urge to manage"**
- 이 병은 '섬기는 리더십'으로 극복
- 겸손, 존경, 신뢰의 분위기 조성에 노력
- 스스로 해결 어려운 관료적 장벽을 제거
- 팀이 합의에 도달하도록 돕기
- 늦게 일하는 동료들에게 저녁 사주기
- 팀의 앞길을 평탄하게 하기 위해 빈틈을 메우고 조언
- 때로는 직접 나서서 일 처리 주저하지 않기
- 기술적, 사회적 건강 모두 챙기기

### The Engineering Manager

- 품목당 한 행 X 재고 단위당 한 행 O
- 10개 단위 예약이면 10개 행을 가짐
- [37signals, DB 기반 부하 분산 방식](https://dev.37signals.com/introducing-solid-queue/)에서 영감
- 예약과 재고를 동일한 DB에 유지하므로 일관성 확보

```sql
-- Start transaction with READ COMMITTED isolation
BEGIN;

-- Select N units, skipping any locked by other transactions
SELECT id
FROM available_units
WHERE shop_id = ?
  AND inventory_item_id = ?
  AND inventory_group_id = ?
ORDER BY shop_id ASC, inventory_item_id ASC, inventory_group_id ASC, id ASC
LIMIT 3
FOR UPDATE SKIP LOCKED;

-- Move selected units to reserved_units table
INSERT INTO reserved_units (unit_id, cart_token, expires_at, ...)

-- Remove from available pool
DELETE FROM available_units
WHERE (shop_id, inventory_item_id, inventory_group_id, id) IN (?, ?, ?, ?);

COMMIT;
```

### 규모 문제 보완

- 하지만, 재고 규모가 크면 문제
- 예컨대, 10개 지역 각각의 5만개 재고에는 50만개 행이 필요
- 예약 조회 시 이 행들을 모두 스캔해야 하므로 속도가 느려짐
  - [MySQL 문서 내용](https://dev.mysql.com/doc/refman/8.4/en/innodb-locking-reads.html) 함께 참고
  - 아마 중복된 값이 여러 행에 걸쳐 있기 때문인 듯
  - 여러 행에 락이 걸려 있으면 일일이 스캔하고 건너 뛰는 비용으로 추정됨
- 그래서 품목/지역 별로 최대 1,000개 행으로 제한된 풀을 유지
- 예약은 이 풀에서 행을 차감 & 재고 보충 프로세스가 재고 원장에서 행을 가져와 풀을 다시 채움
- 그런데 왜 1,000개인가?
- 수요 급증에도 재고 부족 일어나지 않을 만큼 충분히 큼
- 동시에 성능 저하도 초래하지 않는 적절한 수준
- 그래도 만약, 재고 풀이 고갈되면?
