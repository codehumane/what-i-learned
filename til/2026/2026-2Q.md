# 26.04.27

Siemens Health Services가 어떤 지표를 써야 실제로 개선 되는지를 실험한 사례.

https://agilealliance.org/resources/experience-reports/actionable-metrics-siemens-health-services/

- 기존 Agile 지표인 스토리 포인트, velocity로는 실패
- 속도는 좋아보이지만 정작 일은 끝나지 않고, 일정 예측은 어려우며, 동시에 여러 작업 진행으로 병목 증가
- **많이 시작하지만, 완료되는 건 적음**
- 스크럼 중심에서 칸반 기반의 **흐름 중심** 사고로 전환
- 핵심 지표
  - **WIP**: 현재 진행 중인 작업 수
  - **Cycle Time**: 시작 - 완료 시간
  - **Throughput**: 일정 기간 동안 완료한 작업 수
- 특히 중요한 건 WIP 제한
  - Little's Law
  - WIP ↑ → Cycle Time ↓
  - WIP ↓ → Cycle Time ↑
  - **새 일을 시작하기 전에 기존 일을 끝내라**
- 이로 인해 Cycle Time 단축, 예측 가능성 향상, 품질 개선, 처리량 증가
  - velocity 대신 monte carlo 시뮬레이션으로 일정 예측 (85% 확률로 언제 끝날지 판단)
  - 또한 이해관계자에게 스토리 포인트 대신 시간 기반 지표인 Cycle Time으로 커뮤니케이션
- WIP 통해 데이터 기반 의사 결정 가능(병목이 어딘지 보이며, 개발자를 추가할지 테스트를 추가할지 등의 리소스 배분)

## GitHub Developer Productivity at 30 Billion Messages per Day

원문은 [여기](https://thenewstack.io/github-developer-productivity-at-30-billion-messages-per-day/#:~:text=At%20GitHub%2C%20these%20productivity%20metrics%20include%3A)고, [여기](https://github.com/codehumane/what-i-learned/blob/master/document/github-developer-productivity.md)에 정리해서 기록함. 특히 재밌던 부분만 여기서 한 번 더 기록.

- `개발자 경험 = 개발자 생산성 + 개발자가 만들어 낸 영향력 + 개발자 만족도`
- 생산성 측정에 [4가지 DORA 메트릭](https://dora.dev/guides/dora-metrics/) 같은 정성적 지표를 사용
- 더불어, 정성적 지표와 정량적 지표의 혼합인, [SPACE 프레임워크](https://queue.acm.org/detail.cfm?id=3454124&utm_source=the+new+stack&utm_medium=referral&utm_content=inline-mention&utm_campaign=tns+platform)나 [DevEx 메트릭](https://queue.acm.org/detail.cfm?id=3595878&utm_source=chatgpt.com) 함께 사용
- 피드백을 수집하기 위해 몇 주 간격에서 분기별로 다양한 개발자 경험 설문조사 진행
- 인터뷰이의 Aradhya가 정의하는 10x 개발자는 아래와 같음

> 단기적 그리고 장기적으로 직면할 수 있는 문제를 예측하는 비전과 기술 역량을 갖추고, 제품의 경계나 엔지니어링 리더십 경계를 넘어 사용자의 확장성 관점의 문제를 고민할 수 있는 개발자
>
> 10x 개발자는 이런 확장성을 예측하고, 백업과 파트 B를 마련하며, 기술 전략을 세울 수 있어야 함. 이를 통해 unknown unknown을 파악해 나가야 함. 또한 코드 품질을 저하시키지 않으면서도 소비자 수요에 맞춰 확장해 나갈 줄 알아야 함.

- 아래 질문들은 사례들

> - 팀을 어떻게 확장했나요?
> - 팀원들에게 어떤 영향을 미쳤죠?
> - 팀을 긍정적인 방향으로 변화시켰나요?
> - 회사의 문화를 긍정적인 방향으로 변화시켰나요?
