## GitHub’s Platform Engineering Organization

- Aradhya의 팀은 깃헙 인프라 및 플랫폼 엔지니어링 조직의 일부
- 다음의 일을 수평적으로 지원
  - 애플리케이션 팀
  - 코드 검색
  - 깃헙 액션
  - 코파일럿
  - 깃헙 Next R&D 팀
  - Azure 클라우드와의 향후 파트너십
- 플랫폼 팀은 대략 300명
- Aradhya는 그 중 1/3 정도 리딩
- Aradhya 팀은 예전에는 데이터 서비스라고 불림
  - 데이터 파이프라인, 운영의 데이터 측면, 메시징 큐 등 담당
  - 객체 스토리지, 콜드 스토리지, 블롭 스토리지 등 데이터 스토리지 문제 해결뿐만 아니라
  - 데이터 패턴, 차세대 확장성 문제, 생산성 문제 등도 다룸

## Extreme Dogfooding of Developer Experience

- GA는 물론, 알파, 베타 버전 전에도 개발자들이 바로 사용할 수 있게 준비 필요
- 그리고 협업 시에도 항상 생산성과 관련한 질문과 설문을 통해 피드백 수집
- 개발자 경험은 단순히 제품이나 UI 같은 경험이 범위를 벗어남
- `개발자 경험 = 개발자 생산성 + 개발자가 만들어 낸 영향력 + 개발자 만족도`

## How to Measure Collaboration Metrics

- 슬랙, 지라, PR, 문서화 같은 최소한의 도구로는 부족하다고 함
- 동기식 및 비동기식 커뮤니케이션을 폭넓게 고려하라는 정도의 이야기
- 이런 저런 얘기가 나오지만, 구체적인 건 없음

## Measuring Developer Experience at GitHub

- 유명한 [4가지 DORA 메트릭](https://dora.dev/guides/dora-metrics/) 같은 정성적 지표를 사용
- 더불어, 정성적 지표와 정량적 지표의 혼합인, [SPACE 프레임워크](https://queue.acm.org/detail.cfm?id=3454124&utm_source=the+new+stack&utm_medium=referral&utm_content=inline-mention&utm_campaign=tns+platform)나 [DevEx 메트릭](https://queue.acm.org/detail.cfm?id=3595878&utm_source=chatgpt.com) 함께 사용
- 피드백을 수집하기 위해 몇 주 간격에서 분기별로 다양한 개발자 경험 설문조사 진행
  - 응답률이 높은 짧은 설문조사들을 혼합
- 설문조사 외에도, 고객과의 대화, 이벤트 기간 동안의 평점 시스템 등을 통해 지속적 수집
- 깃헙에서 사용하는 생산성 지표는 아래와 같음

```
- 사용자로부터 피드백을 받는 데 얼마나 시간이 걸립니까?
- 비동기 통신은 어떤 모습일까요?
- 회의나 방해 없이 한 번에 얼마나 오랫동안 집중해서 일할 수 있습니까?
- 새로운 문제에 대한 해결책을 개발하는 데 얼마나 많은 시간을 투자하고 계십니까?
- 코딩이나 버그 수정에 얼마나 많은 시간을 소비하시나요?
- 보안 또는 취약점 문제를 얼마나 자주 접하십니까?
- 신제품 출시나 업데이트 기간 동안 자기계발에 얼마나 많은 시간을 투자하시나요? 다른 사람들을 멘토링하고 역량을 향상시키는 데는 얼마나 많은 시간을 할애하시나요?
- 자동화 테스트 작성에 얼마나 많은 시간이 소요되나요?
- 컨텍스트를 얼마나 자주 전환하시나요?
- 얼마나 생산적이라고 느끼시나요?
```

## Global Scale Demands Unconventional Developer Productivity

- 위 지표들은 10x 개발자 정신을 함양하기 위함
- Aradhya가 정의하는 10x 개발자는 아래와 같음

> 단기적 그리고 장기적으로 직면할 수 있는 문제를 예측하는 비전과 기술 역량을 갖추고, 제품의 경계나 엔지니어링 리더십 경계를 넘어 사용자의 확장성 관점의 문제를 고민할 수 있는 개발자

- 코파일럿의 경우 1.5만명 사용자를 예상했으나 곧 1.15억 사용자를 마주하며, 신뢰성과 보안을 고려한 확장성을 갖추는 게 매우 중요했기 때문
- 모든 깃헙 제품들은 반드시 확장 가능해야 함

> 10x 개발자는 이런 확장성을 예측하고, 백업과 파트 B를 마련하며, 기술 전략을 세울 수 있어야 함. 이를 통해 unknown unknown을 파악해 나가야 함. 또한 코드 품질을 저하시키지 않으면서도 소비자 수요에 맞춰 확장해 나갈 줄 알아야 함.

- MS에서는 일반적으로 누구나 10x 개발자가 될 가능성이 있음
- 이를 위해 독창적 사고(out-of-the-box thinking) 발휘와, 문화와 협업을 중시해야 함

> - 팀을 어떻게 확장했나요?
> - 팀원들에게 어떤 영향을 미쳤죠?
> - 팀을 긍정적인 방향으로 변화시켰나요?
> - 회사의 문화를 긍정적인 방향으로 변화시켰나요?

- 깃헙은 자동화된 피드백도 많이 수집
- 피드백 루프를 강화할 방법을 항상 고민
- 예를 들어, 유효성 검사와 규정 준수 확인이나
- 개발자가 가장 비생산적인 부분을 파악하도록 돕는 것이나
- 고객 피드백이 개발자에게 전달되게 해서 피처 리퀘스트에 포함되게 하는 것(이 부분 강조함)
- 이들 측정은 익명의 인증된 설문이나, 이슈, PR 등의 혼합으로 이뤄짐
- 설문이 너무 많으면 답변을 거부하므로 내부의 분석 자료와 마케팅 퍼널도 살핌
- 이런 모든 결과는 아래와 같은 형태로 전달되고 누구나 볼 수 있음

> "“This is the percentage of time you’ve spent on X. And we improve that with Y. And then ask for feedback again.”"
